# Extensions

Most users do not need the extension API. The `ExtensionRegistry` is the entry
point for advanced callers who want to add custom lint rules or transforms for
a single execution without modifying Scribpy internals. HTML builds also expose
an internal code-block plugin hook for block renderers such as PlantUML or
future Mermaid support.

---

## `ExtensionRegistry`

```python
from scribpy.extensions import ExtensionRegistry
```

### Class methods

#### `ExtensionRegistry.native() -> ExtensionRegistry`

Return a new registry pre-loaded with all built-in Scribpy lint rules and
transforms. This is the correct starting point for any custom registry.

```python
registry = ExtensionRegistry.native()
```

### Instance methods

#### `with_lint_rule(rule: LintRule) -> ExtensionRegistry`

Return a new registry with one additional lint rule appended after the built-ins.
The original registry is not mutated.

#### `with_markdown_transform(transform: Transform) -> ExtensionRegistry`

Return a new registry with one additional Markdown transform appended after the
built-ins. The original registry is not mutated.

#### `with_code_block_plugin(plugin: CodeBlockPlugin) -> ExtensionRegistry`

Return a new registry with one additional fenced-code-block plugin appended
after Scribpy's built-in plugins. A plugin can detect a Markdown fence language,
run an optional preflight check, emit assets, and replace the original blocks in
the transformed output.

### Properties

| Property | Type | Description |
|---|---|---|
| `lint_rules` | `tuple[LintRule, ...]` | All lint rules in execution order. |
| `markdown_transforms` | `tuple[Transform, ...]` | All Markdown transforms in execution order. |
| `html_transforms` | `tuple[Transform, ...]` | All HTML transforms in execution order. |
| `code_block_plugins` | `tuple[CodeBlockPlugin, ...]` | Additional fenced-code-block plugins in execution order. |

## Writing a custom lint rule

A lint rule is any object that satisfies the `LintRule` protocol:

```python
from scribpy.lint import LintRule, LintContext
from scribpy.model import Diagnostic
from dataclasses import dataclass

@dataclass(frozen=True)
class NoTodoRule:
    code: str = “CUSTOM001”

    def run(self, context: LintContext) -> tuple[Diagnostic, ...]:
        diagnostics = []
        for doc in context.documents:
            if “TODO” in doc.source:
                diagnostics.append(Diagnostic(
                    severity=”warning”,
                    code=self.code,
                    message=”Document contains a TODO comment.”,
                    path=doc.path,
                    hint=”Resolve or remove TODO before publishing.”,
                ))
        return tuple(diagnostics)
```

## Writing a custom transform

A transform is a callable with the signature
`(TransformContext) -> TransformResult`:

```python
from scribpy.transforms import TransformContext, TransformResult, TransformedDocument

def add_footer_transform(context: TransformContext) -> TransformResult:
    updated = []
    for doc in context.transformed_documents:
        content = doc.content + “\n\n---\n*Generated by Scribpy.*\n”
        updated.append(TransformedDocument(
            relative_path=doc.relative_path,
            content=content,
            source_document=doc.source_document,
        ))
    return TransformResult(documents=tuple(updated))
```

## Using a custom registry

Pass the registry to any `scribpy.core` function via the `registry=` keyword:

```python
from scribpy.extensions import ExtensionRegistry
from scribpy.core import lint_project, build_project

registry = (
    ExtensionRegistry.native()
    .with_lint_rule(NoTodoRule())
    .with_markdown_transform(add_footer_transform)
)

# Lint with custom rules
result = lint_project(“my-project”, registry=registry)

# Build with custom transforms
result = build_project(“my-project”, target=”markdown”, registry=registry)
```

!!! note
    The `registry=` parameter is available on `scribpy.core` functions, not on
    the top-level `scribpy.*` convenience functions. Use `scribpy.core` directly
    when you need this degree of control.

---

## When to use the extension API

- Add a project-specific lint rule (e.g. enforce a custom style guide);
- Add a Markdown post-processing transform (e.g. inject a footer, add metadata);
- Compose built-in behaviour with local extensions in CI automation or tests.

**Keep extensions focused.** Each rule or transform should do one thing.
Scribpy’s immutable registry pattern ensures that custom registries never affect
unrelated builds.
